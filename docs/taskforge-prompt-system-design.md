# TaskForge Prompt 体系优化方案

## 1. 背景与目标

当前 `TaskForge` 已经具备较清晰的主线：

- 将需求、设计、接口、图标、项目上下文统一录入
- 将需求拆成可执行任务
- 为任务生成标准化 Prompt
- 调用本地 Agent 在受控边界内执行

这条主线方向正确，但如果目标是让系统长期稳定、可控、可复用地交付代码，仅靠“生成一段标准 Prompt”还不够。真正需要沉淀的核心能力，不是 Prompt 文案本身，而是：

- 结构化规格如何表达
- 任务如何切分为最小可验证单元
- 不同类型任务如何装配不同 Prompt
- 执行前后如何做门控与校验
- 如何度量 Prompt 是否真的在持续变好

因此，TaskForge 下一阶段应从“Prompt 生成”升级为“可执行规格编译”。

一句话概括：

**Prompt 不应是核心资产，结构化规格与可验证执行协议才是核心资产；Prompt 只是中间产物。**

---

## 2. 现阶段 Prompt 方案的主要问题

从当前文档描述看，现有方案已经强调了边界和复用，但仍存在几个常见风险。

### 2.1 单段 Prompt 承载过多职责

如果一个任务只生成一段较长 Prompt，往往会把以下内容混在一起：

- 任务目标
- 项目上下文
- 设计与接口约束
- 禁止事项
- 执行步骤
- 输出格式

结果通常是 Prompt 很长，但信息层次不清。模型可能看到“都说了”，但并没有真正抓住优先级。

### 2.2 “优先复用”仍然是软约束

如果 Prompt 只是写“优先复用现有组件、Token、API 封装”，模型很容易口头遵守、实际重写。  
软约束无法稳定阻止重复造轮子。

### 2.3 缺少“信息不足时必须澄清”的规则

很多任务失败，不是因为模型不会写，而是输入不完整。若系统默认“继续做”，就容易出现：

- 自行猜测接口字段
- 自行补全交互状态
- 自行创建与项目风格不一致的新组件
- 为了完成任务而越界改动

### 2.4 任务拆分与验收没有完全绑定

任务如果只具备“要做什么”，但没有“如何验证完成”，会导致任务看起来拆小了，实际仍然模糊。

### 2.5 缺少 Prompt 质量反馈闭环

如果没有对执行成功率、返工率、越界率做记录，Prompt 迭代就只能靠经验优化，无法形成真正的产品能力。

---

## 3. 目标形态：从 Prompt 模板升级为执行编排系统

建议将 TaskForge 的执行链路明确定义为五层：

1. `规格层`：描述需求、验收、边界、资源与禁止事项
2. `计划层`：结合仓库扫描结果，收敛到可落地实现空间
3. `任务层`：将计划拆解为最小可验证单元
4. `执行层`：将结构化任务编译为对应类型的 Prompt
5. `校验层`：对执行结果进行结构化核对与反馈归因

这五层中，真正需要长期沉淀的是前四层的数据结构和第五层的反馈机制，而不是某一版 Prompt 文案。

---

## 4. Prompt 体系总体架构

## 4.1 分层 Prompt 架构

每个任务的最终 Prompt 应由多个层级装配而成，而不是一次性拼成长文本。

### A. System 层

定义 Agent 的通用身份与不可突破的底线：

- 当前角色是“仓库内受控执行代理”
- 只能在允许范围内读取和修改文件
- 信息不足时不得自行猜测核心业务规则
- 超过门槛的改动必须先提出确认
- 必须先分析、后计划、再执行、再总结

这一层应尽量稳定，不随任务波动。

### B. Project 层

描述当前仓库的固定上下文：

- 技术栈
- 项目目录结构
- UI 组件库
- 主题与 Token 体系
- API 调用与数据层规范
- 测试、构建、Lint 约束
- 常见禁改目录或高风险区域

这一层按项目维度复用，同一项目下尽量不要每个任务重复生成。

### C. Task 层

只描述当前任务的核心差异信息：

- 当前要完成什么
- 本任务关联哪些资源
- 本任务允许修改哪些文件
- 本任务禁止碰哪些区域
- 验收标准是什么
- 若信息不足，需要提出哪些澄清问题

### D. Output 层

约束模型输出格式，避免自由发挥：

- 先输出实施计划
- 列出涉及文件、组件、函数
- 给出复用点选择依据
- 给出风险与待确认点
- 执行完成后输出变更摘要

### E. Verification 层

要求模型在执行后进行自检：

- 是否超范围改动
- 是否按约定复用了现有实现
- 是否存在未披露假设
- 是否满足验收标准
- 是否仍有人工验证项

## 5. 任务数据结构建议

要让 Prompt 稳定，前提是任务结构稳定。建议将 TaskForge 的任务对象收敛为以下核心字段。

```json
{
  "id": "task-001",
  "type": "ui_implementation",
  "title": "实现用户列表页面筛选区",
  "goal": "根据设计稿实现筛选区，并接入现有查询参数状态管理",
  "why": "补齐用户列表页面的核心操作入口",
  "acceptance_criteria": [
    "UI 与 Figma 主体结构一致",
    "复用现有表单组件和主题 Token",
    "筛选参数与现有列表查询逻辑打通",
    "不修改无关页面结构"
  ],
  "resources": {
    "figma": ["figma-node-1"],
    "api_contracts": ["user-list-query"],
    "assets": [],
    "references": ["src/pages/user-list", "src/components/SearchForm"]
  },
  "allowed_changes": {
    "files": [
      "src/pages/user-list/**",
      "src/components/**"
    ],
    "new_files": [
      "src/pages/user-list/components/*.tsx"
    ]
  },
  "forbidden_changes": {
    "files": [
      "src/core/**",
      "src/theme/foundation/**"
    ],
    "actions": [
      "不要修改全局主题变量",
      "不要重构与本任务无关的 API 层"
    ]
  },
  "reuse_requirements": [
    "优先复用现有表单容器组件",
    "优先复用现有查询参数同步逻辑",
    "优先复用现有按钮与输入框"
  ],
  "clarification_rules": [
    "若 Figma 未体现交互状态，不得自行扩展业务逻辑",
    "若现有仓库中存在多个候选组件，需先列出对比与选择依据"
  ],
  "verification": {
    "manual_checks": [
      "人工确认筛选项文案",
      "人工确认暗色模式下视觉效果"
    ],
    "automated_checks": [
      "lint",
      "targeted test if available"
    ]
  }
}
```

这个结构的重点不是字段多，而是把“目标、边界、资源、验收、澄清、校验”拆开，避免混写。

## 6. 任务分型建议

TaskForge 不应只维护一个通用 Prompt 模板，而应至少按任务类型分型。

建议首批支持以下类型：

### 6.1 `ui_implementation`

适用于按设计稿实现页面或模块。

关注重点：

- Figma 对齐
- Token 与组件复用
- 响应式与状态完整性
- 不凭空发明交互逻辑

### 6.2 `api_integration`

适用于页面接入接口、补齐请求参数、联调数据流。

关注重点：

- 接口契约一致性
- 参数映射与错误处理
- 复用既有 API 封装
- 不绕开统一数据层

### 6.3 `page_refactor`

适用于在既有页面上做结构调整。

关注重点：

- 最小可控改动
- 向后兼容
- 明确列出受影响模块
- 防止顺手重构无关逻辑

### 6.4 `component_extraction`

适用于抽离通用组件、复用逻辑。

关注重点：

- 抽象边界是否合理
- 是否真的具备复用价值
- 是否破坏现有调用方式
- 是否引入额外复杂度

### 6.5 `bug_fix`

适用于修复问题。

关注重点：

- 根因定位
- 最小变更面
- 回归风险
- 是否需要补测试

### 6.6 `backend_implementation`

适用于新增后端接口、数据结构、服务层逻辑。

关注重点：

- 契约一致
- 数据模型与已有架构一致
- 权限、事务、错误处理
- 与已有模块复用关系

---

## 7. 每类 Prompt 的共用槽位

不管任务类型如何变化，建议所有 Prompt 都统一保留以下槽位：

1. `角色`
2. `任务目标`
3. `成功标准`
4. `任务依据`
5. `允许修改范围`
6. `禁止事项`
7. `复用要求`
8. `澄清触发规则`
9. `执行步骤要求`
10. `输出格式`
11. `执行后自检`

这样可以保证：

- 模板之间具有统一骨架
- 任务类型差异只体现在具体内容和少量附加槽位
- 后续更容易做质量评估和 A/B 对比

---

## 8. 推荐的 Prompt 骨架

以下骨架适合作为 TaskForge 的通用编译目标，再按任务类型做增减。

```text
[角色]
你是当前仓库中的受控实现代理，只能在给定任务范围内工作。

[任务目标]
完成以下任务：
<goal>

[成功标准]
<acceptance_criteria>

[任务依据]
需求摘要：
<spec summary>

设计依据：
<figma bindings or empty>

接口依据：
<api contracts or empty>

项目参考实现：
<relevant files/components>

[允许修改范围]
允许修改：
<allowed files>

允许新增：
<allowed new files>

[禁止事项]
禁止修改：
<forbidden files>

禁止行为：
<forbidden actions>

[复用要求]
优先复用以下已有能力：
<reuse requirements>

[澄清触发规则]
若出现以下情况，先输出问题，不要直接执行：
<clarification rules>

[执行要求]
1. 先分析现有实现、候选复用点与影响范围
2. 输出实施计划
3. 列出将修改的文件、组件、函数
4. 说明为什么选择这些改动点
5. 在无阻塞信息缺口时执行
6. 完成后输出改动摘要与风险

[输出格式]
- 方案摘要
- 复用点候选与最终选择
- 改动文件清单
- 风险点
- 待确认问题
- 最终改动摘要

[执行后自检]
- 是否严格遵守允许修改范围
- 是否说明未复用已有实现的原因
- 是否存在未说明假设
- 是否满足成功标准
- 是否仍有需要人工验证的项目
```

## 9. 关键强化点：把软约束改成硬门控

下面几项尤其建议从自然语言建议，升级成“必须输出结构化结果”。

### 9.1 复用不再是建议，而是必答项

执行计划里必须包含：

- `候选复用点`
- `最终复用对象`
- `未复用原因`
- `若新增组件，新增的必要性`

如果这些字段为空，不进入执行。

### 9.2 计划不再只是“怎么做”，而要“说明依据”

执行计划里必须包含：

- 本次改动为何落在这些文件
- 这些文件与任务目标的关系
- 是否存在更小改动面
- 为什么当前方案与项目现有模式一致

### 9.3 信息不足时必须停下

出现以下情况必须进入澄清分支：

- 设计稿缺少关键状态
- 接口契约缺失字段或返回结构
- 多个候选组件无法判断应复用哪一个
- 改动范围超过任务允许上限
- 现有实现与需求存在明显冲突

### 9.4 执行后必须做自检与归因

结果输出不应只包含“改了什么”，还应包含：

- 哪些验收条件已满足
- 哪些条件仍需人工确认
- 是否出现额外假设
- 是否发现当前规格本身存在缺口

---

## 10. 校验体系设计

TaskForge 如果要真正把 Prompt 做成产品能力，必须有执行后校验机制。建议校验分三层。

### 10.1 结构化自检

由执行 Agent 自己按固定字段输出：

- 是否越界修改
- 是否遵循设计与接口
- 是否复用已有实现
- 是否存在风险

### 10.2 规则校验

由系统在执行结果外部检查：

- 改动文件是否超范围
- 是否输出了必填字段
- 是否存在未绑定资源却被引用的内容
- 是否跳过了先计划后执行的流程

### 10.3 人工校验

对暂时难以自动判断的内容保留人工确认：

- 视觉细节是否与设计稿完全一致
- 业务文案是否准确
- 复杂交互与边界状态是否满足预期

---

## 11. Prompt 质量评估指标

建议在产品中沉淀以下指标，用来反向指导模板优化。

### 11.1 执行质量指标

- 首次执行成功率
- 首次执行后仍需返工的比例
- 平均返工轮次
- 平均人工介入次数
- 超范围修改率

### 11.2 任务质量指标

- 任务是否具备完整验收标准
- 任务是否触发澄清
- 澄清后成功率是否提升
- 不同任务类型的完成率差异

### 11.3 复用质量指标

- 组件复用率
- Token 复用率
- API 封装复用率
- 新建组件后被再次复用的比例

### 11.4 Prompt 模板质量指标

- 不同模板在不同任务类型上的成功率
- Prompt 长度与成功率关系
- 约束数量与执行稳定性关系
- 澄清规则触发频率与返工率关系

## 12. 建议的数据模型

为了让系统真正具备演进能力，建议至少固化以下对象。

### 12.1 `Spec`

描述需求与约束本身：

- 背景
- 目标
- 范围
- 验收标准
- 资源绑定
- 禁止事项
- 待确认问题

### 12.2 `ExecutionPlan`

描述在当前仓库里的落地路径：

- 涉及模块
- 候选复用点
- 风险点
- 建议拆分方式
- 是否需要人工确认

### 12.3 `Task`

描述最小可验证单元：

- 类型
- 目标
- 允许改动范围
- 验收标准
- 绑定资源
- 澄清规则

### 12.4 `PromptPackage`

描述最终交给 Agent 的执行包：

- system_prompt
- project_context
- task_prompt
- output_contract
- verification_contract

### 12.5 `ExecutionResult`

描述执行产物与反馈：

- 修改文件
- 结构化摘要
- 自检结果
- 风险与未决问题
- 自动校验结果
- 人工确认结果

---

## 13. 建议的落地顺序

不建议一次把所有能力做完，建议分三阶段推进。

### 第一阶段：先把 Prompt 从“文本模板”升级为“分层模板”

优先做：

- 定义 System / Project / Task / Output / Verification 五层结构
- 收敛统一任务数据结构
- 为三类高频任务建立独立模板

建议首批任务类型：

- `ui_implementation`
- `api_integration`
- `bug_fix`

### 第二阶段：引入硬门控与澄清分支

优先做：

- 复用点输出必填
- 改动范围检查
- 信息不足时停止执行
- 计划输出结构化

### 第三阶段：做反馈闭环与指标看板

优先做：

- 执行结果结构化落库
- 统计模板成功率
- 标记返工原因
- 形成 Prompt A/B 优化基础

---

## 14. 可直接用于产品描述的话术

如果需要把这套优化讲给管理层或研发团队，可以统一成以下表达：

- TaskForge 的核心不再是“帮用户拼一段 Prompt”，而是把需求编译成可执行规格
- Prompt 只是执行载体，真正稳定的是规格结构、任务边界和验证协议
- 我们要做的不是更长的 Prompt，而是更少歧义、更强约束、更可验证的执行编排
- 同一种任务应由同一种模板驱动，不同类型任务应有不同执行协议
- 真正的优化目标不是模型回答更漂亮，而是首轮成功率更高、返工更少、复用更强、越界更少

---

## 15. 最终建议

对 TaskForge 来说，Prompt 优化最重要的不是继续打磨提示词文案，而是完成以下四个升级：

1. 从单段 Prompt 升级为分层 Prompt 装配
2. 从通用模板升级为任务分型模板
3. 从软约束升级为结构化硬门控
4. 从经验优化升级为指标驱动优化

当这四点完成后，TaskForge 的核心竞争力就不再只是“会调用 Agent”，而是具备了一套可积累、可扩展、可复制的 AI 执行编排能力。
